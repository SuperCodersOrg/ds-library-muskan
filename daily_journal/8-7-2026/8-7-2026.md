# Section 1 — Specific Bug / Confusion

Today's biggest confusion was understanding why two `Pair` objects appeared during insertion.

Inside `set()`, I created a temporary `Pair` object before calling `insertBack()`.

When the linked-list node was created, another `Pair` object was copy-constructed inside the node.

Initially, I thought both objects remained in memory and that my implementation was wasting memory.

After tracing constructor and destructor calls, I realized the object inside `set()` is only a temporary stack object. Once insertion completes, it is destroyed automatically, while the heap-allocated object inside the node continues to exist.

This helped me understand temporary object lifetime much better.

---

# Section 2 — Failed Attempt

After implementing the first version of `HashMap`, I wrote a custom Rule of Three because I assumed every container class should define:

- destructor
- copy constructor
- copy assignment operator

While tracing destruction, I observed that the compiler-generated destructor automatically destroys member objects.

The destruction sequence is:

```
HashMap
    ↓
DynamicArray
    ↓
Each LinkList
    ↓
Each Node
    ↓
Stored Pair Object
```

Likewise, copying the `HashMap` automatically invokes `DynamicArray`'s copy constructor, which already performs deep copying.

Since `HashMap` owns no raw heap memory, my custom Rule of Three was unnecessary, so I removed it.

I also upgraded the hash function from simple division hashing to MurmurHash to improve key distribution and reduce clustering.

# Section 4 — Learning Reflection

Today reinforced two important C++ concepts.

First, I learned to distinguish between **temporary objects** and **persistent objects**. A temporary `Pair` created in `set()` only exists long enough to initialize the `Pair` stored inside the linked-list node. Understanding this clarified how constructors and destructors work together during insertion.

Second, I learned that the **Rule of Three depends on ownership, not composition**. Although `HashMap` contains objects that manage heap memory, it does not directly own any raw resources. The compiler-generated destructor, copy constructor, and copy assignment operator work correctly because `DynamicArray` and `LinkList` already manage their own resources. This insight helped simplify the implementation while keeping it correct.

Finally, replacing division hashing with MurmurHash helped me understand that the quality of a hash function has a direct impact on collision frequency and overall hash table performance. This completed the transition from a basic integer-specific implementation to a reusable generic `HashMap`.# Day Journal
